# 第12回課題：遺伝的アルゴリズムの並列化

## 1. 目的

本課題では、2次元Rastrigin関数を最小化する遺伝的アルゴリズム（GA）を対象に、並列実行できる処理を特定し、Worker数を変えたときの実行時間から並列化の効果を考察する。

目的関数は次式である。

$$
f(x,y)=20+x^2+y^2-10\{\cos(2\pi x)+\cos(2\pi y)\}
$$

この関数は多数の局所最小解を持ち、大域的最小解は $(x,y)=(0,0)$、最小値は $0$ である。本実装は最小化問題なので、目的関数値が小さい個体ほど良い個体として扱う。

## 2. 並列実行できる処理

| 処理 | 並列化 | 理由・依存関係 |
|---|---|---|
| 個体群の初期化 | 可能 | 各個体を独立に生成できる。ただし処理が軽いため効果は小さい |
| 各個体の適応度評価 | 最も適している | 各個体のRastrigin関数値は他個体の計算結果に依存しない |
| トーナメント選択 | 原理上可能 | 各トーナメントは独立だが、全個体の適応度評価完了を待つ必要がある |
| 交叉 | 原理上可能 | 親の組ごとに子を独立して生成できる |
| 突然変異 | 原理上可能 | 個体・遺伝子ごとに独立して処理できる |
| 最良値・平均値の集計 | 一部可能 | 最小値や平均値を求めるリダクション処理と同期が必要 |
| エリート保存 | 一部可能 | 子個体の評価は並列化できるが、最悪個体の決定と置換は全体結果を必要とする |
| 世代間処理 | 不可 | 次世代は前世代の評価・選択・交叉・突然変異の完了後にしか生成できない |

本プログラムで中心となる並列化箇所は、個体ごとの適応度評価である。各個体の計算は独立しているため、次のように`pool.map()`で個体群をWorkerへ分配している。

```python
with multiprocessing.Pool(processes=self.num_workers) as pool:
    fitnesses = pool.map(rastrigin, population)
```

一方、親の選択には全個体の適応度が必要であり、次世代は現在の世代の処理が終わるまで生成できない。そのため、適応度評価後、選択後、エリート置換後には同期が必要で、異なる世代を同時には進められない。

## 3. 実験方法

配布Notebookの条件を使用した。

| 項目 | 設定 |
|---|---:|
| 個体数 | 100 |
| 世代数 | 80 |
| 探索範囲 | $[-5.12, 5.12]$ |
| 突然変異率 | 0.15 |
| 突然変異の標準偏差 | 0.3 |
| Worker数 | 1、2、4、8 |
| 乱数シード | 42 |
| 計時方法 | `time.perf_counter()` |

Worker数 $p$ の実行時間を $T_p$ とし、速度向上率 $S_p$ と並列効率 $E_p$ を次式で求めた。

$$
S_p=\frac{T_1}{T_p}, \qquad E_p=\frac{S_p}{p}
$$

Worker数を増やすほど実行時間が長くなり、8 Workerでは1 Workerの約5.3倍の時間を要した。したがって、今回の条件では並列化は高速化につながらず、むしろ逆効果となった。

## 5. 考察

### 5.1 並列化で遅くなった理由

Rastrigin関数の評価は、2変数に対する四則演算と余弦計算だけであり、1個体あたりの計算量が非常に小さい。一方、`multiprocessing`には次の費用が発生する。

- Workerプロセスの起動と終了
- 個体データのシリアライズと転送
- Workerへのタスク割り当て
- 結果の回収と並び替え
- プロセス間通信とコンテキストスイッチ
- 全Workerの終了を待つ同期

さらに、配布コードは適応度評価のたびに`Pool`を新しく生成している。各世代では現在の個体群の評価と、エリート保存のための子個体群の評価を行い、最後にも評価するため、80世代の1回の実行で合計161回の`Pool`生成・終了が発生する。この起動・終了コストが目的関数の計算時間を大きく上回ったことが、Worker数を増やすほど遅くなった主因と考えられる。

### 5.2 逐次処理と同期の影響

並列化可能なのは主に適応度評価であり、選択、履歴の更新、エリートの置換、世代更新などには逐次処理が残る。並列化可能割合を $P$、Worker数を $p$ とすると、オーバーヘッドを無視した理論速度向上率はAmdahlの法則により次式で表される。

$$
S(p)=\frac{1}{(1-P)+P/p}
$$

Worker数を増やしても逐次部分は短縮されず、実際には通信や同期のオーバーヘッドも加わる。そのため、Worker数に比例した高速化は得られない。

### 5.3 並列化が有効になる条件

今回の結果は、GAの並列化自体が無効であることを意味しない。画像認識モデルの学習・推論、物理シミュレーション、複雑な制約判定など、1個体の評価が十分に重い問題では、プロセス間通信の費用より評価時間の短縮効果が大きくなり、並列化が有効になる。

実装面では、次の改善が考えられる。

1. `Pool`を世代ごとに作り直さず、GAの開始時に一度だけ生成して世代間で再利用する。
2. 複数個体をまとめてWorkerへ渡し、通信回数を減らす。
3. CPUの物理コア数を超えてWorkerを増やさない。
4. 同じ条件を複数回測定し、中央値とばらつきで比較する。
5. 適応度評価が軽い場合は、NumPyによるベクトル化を優先する。

## 6. 結論

遺伝的アルゴリズムでは、個体ごとの適応度評価、親の組ごとの交叉、個体ごとの突然変異を並列化できる。特に適応度評価は個体間の依存関係がなく、データ並列化に適している。一方、選択、集計、エリート保存、世代更新には全体結果が必要であり、同期点と逐次処理が残る。

今回の実験では、1 Workerの2.892秒に対し8 Workerでは15.291秒となり、Worker数を増やすほど遅くなった。目的関数の計算が軽いことに加え、適応度評価のたびに`Pool`を生成する実装のため、プロセス生成・通信・同期の費用が計算時間の削減量を上回ったためである。並列化を有効にするには、Poolの再利用やタスクのまとめ処理を行い、1個体あたりの評価が十分に重い問題へ適用する必要がある。
